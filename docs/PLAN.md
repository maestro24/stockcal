# 기획서 #4 — 공모주 / 배당 캘린더 (가칭: 주식달력)

> 작성일: 2026-07-10 | 상태: Draft v1 | 배포: GitHub Pages/Vercel 정적 + GitHub Actions 일일 갱신, 비용 0원

## 1. 문제 정의

- 공모주 청약 일정·배당락일 정보는 수요가 크고 검색량이 꾸준하지만("공모주 일정", "배당락일", "이번주 공모주"), 기존 정보원은 (a) 광고 범벅 블로그, (b) 로그인 필요한 증권사 앱, (c) 표 형태로 흩어진 커뮤니티 글로 파편화되어 있다.
- 이 시장은 콘텐츠가 "데이터 그 자체"라 매일 자동 갱신만 되면 사람이 글을 쓸 필요가 없다.
- 금융 니치는 AdSense RPM 최상위권 — 동일 트래픽 대비 수익이 게임/유틸의 수 배.

## 2. 목표 & 성공 지표

| 지표 | 3개월 | 6개월 |
|------|-------|-------|
| 월 방문자 (SEO) | 3,000 | 20,000 |
| 구글 상위 10위 키워드 | 5개 | 30개 |
| 데이터 갱신 성공률 (Actions) | 99% | 99% |
| 월 수익 (AdSense+제휴) | 승인 | 30만 원+ |

## 3. 타겟 사용자

- 1차: 공모주 균등배정 노리는 소액 투자자 (전 연령, 검색 유입)
- 2차: 배당주 투자자 (배당락일·지급일 캘린더 수요)
- 사용 맥락: "이번 주에 청약할 거 있나?" — 주 1–2회 확인, 모바일 위주

## 4. 핵심 기능 (MoSCoW)

### Must
- **공모주 캘린더**: 종목별 청약 시작/종료일, 상장일, 공모가 밴드, 주관사, 균등/비례 정보. 월간 캘린더 뷰 + 리스트 뷰
- **종목 상세 페이지** (`/ipo/2026/종목명`): 일정 + 기본 정보. 상장 후 결과(따상 여부) 업데이트 → 아카이브가 SEO 자산으로 축적
- **배당 캘린더**: 국내 주요 종목 + 미국 인기 배당주 배당락일/지급일
- GitHub Actions 크론 (1일 1회): 데이터 수집 → JSON 생성 → 커밋 → 자동 재배포

### Should
- 청약 증거금 계산기 (균등 최소 증거금), 배당 수익률 계산기 — 계산기 = 추가 SEO 페이지
- iCal 내보내기 (.ics 파일 정적 생성) — "구글 캘린더에 추가" 차별화 기능

### Could
- 지난 공모주 성과 통계 페이지 (연도별 상장일 수익률 표)
- RSS 피드

### Won't (서버 필요 → 제외)
- 알림(카톡/이메일), 로그인, 개인화

## 5. 데이터 소스 (핵심 리스크 영역)

| 데이터 | 후보 소스 | 방식 | 비용 |
|--------|----------|------|------|
| 공모주 일정 | KIND(kind.krx.co.kr) 공시, 38커뮤니케이션(참고), DART OpenAPI | Actions에서 Python 스크레이핑/API | 0원 (DART API 무료 키) |
| 배당 (국내) | DART 배당 공시, KRX 정보데이터시스템 | 동일 | 0원 |
| 배당 (미국) | FinanceDataReader / yfinance | 동일 | 0원 |

**원칙**: 공공 공시 데이터(KIND/DART)를 1차 소스로. 사설 사이트 스크레이핑은 법적·안정성 리스크 있어 참고용만. 이 레포에 이미 FinanceDataReader + Actions 크론 패턴 존재(XRP 모니터링 봇) → 재사용.

## 6. 기술 스택 & 아키텍처

```
[GitHub Actions cron 06:00 KST]
   → Python 수집 스크립트 (DART/KRX/FDR)
   → data/ipo.json, data/dividend.json 생성
   → git commit & push
   → Vercel/Pages 자동 재빌드
[정적 사이트]
   → Astro (또는 Next.js SSG) — JSON 읽어 페이지 정적 생성
   → 종목별 개별 HTML = SEO 페이지 자동 양산
```

- Astro 추천: JSON → 수백 개 정적 페이지 생성에 최적, JS 최소 출력
- 스키마 마크업: `Event`(청약 일정), `FAQPage` — 리치 결과 노출
- 데이터 검증: 수집 스크립트에 스키마 검증 + 실패 시 이전 JSON 유지(사이트 절대 안 깨짐) + Actions 실패 알림

## 8. 수익화

1. AdSense (금융 RPM 프리미엄)
2. 증권사 계좌개설 제휴 (뱅크샐러드/핀크 스타일 제휴 링크 — 공모주 = 계좌개설 수요와 직결, 건당 수익 큼)
3. 도서 제휴 (쿠팡파트너스: 공모주/배당 투자서)

## 9. SEO 전략

- 타깃 키워드: "공모주 일정"(head), "{종목명} 청약"(수백 개 롱테일 자동 생성), "{종목명} 배당락일", "이번주 공모주"
- 종목 페이지는 상장 후에도 삭제하지 않음 — "{종목} 공모가", "{종목} 상장일 주가" 아카이브 검색 수요 흡수
- 사이트맵 자동 생성, 매일 갱신되는 사이트 = 크롤링 빈도 상승

## 13. 핵심 원칙

1. **데이터가 곧 콘텐츠**: 사람이 글을 쓰지 않는다. 수집→검증→렌더 파이프라인이 제품이다.
2. **절대 깨지지 않는 사이트**: 수집 실패 시 이전 데이터 유지. 빈 데이터면 "준비 중" 상태를 정직하게 표시. 잘못된 데이터 노출 > 데이터 없음.
3. **서버 제로**: GitHub Actions 크론이 유일한 "백엔드". 결과물은 정적 JSON + HTML.
4. **공공 데이터 우선**: DART/KIND 공시 기준. 사설 사이트 스크레이핑은 금지. 데이터엔 항상 기준일·출처·면책 표기.
5. **가짜 데이터 금지**: 실제처럼 보이는 예시 종목 절대 노출 금지. 샘플은 명시적 '예시' 라벨 + 개발 모드에서만.
6. **모바일 우선**: "이번 주에 청약할 거 있나?" 를 3초 안에 답하는 화면.
7. **점진 개선**: 단계마다 시각·성능·느낌 +12%, 매 단계 브라우저 실검증.
